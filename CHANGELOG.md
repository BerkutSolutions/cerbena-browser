# Журнал изменений

## 1.0.2 — Обновление интерфейса и импорта расширений

### Основное

- Версия продукта обновлена до `1.0.2` в desktop-приложении, документации, runtime-метаданных и релизных точках.
- Импорт архивов расширений стал устойчивее: если в ZIP лежит несколько отдельных расширений по папкам, каждое из них импортируется как самостоятельный элемент библиотеки.
- Сценарий одиночного импорта `.zip`, `.xpi` и `.crx` сохранен без изменения пользовательского потока.

### Повышение безопасности

- Десктопный апдейтер теперь требует `checksums.txt` и отдельную подпись `checksums.sig`, проверяет подпись контрольных сумм и сверяет SHA-256 скачанного релизного артефакта до staging и apply.
- Release pipeline теперь выпускает подписанный checksum-артефакт, чтобы клиент обновления не доверял неподтвержденным релизным файлам.
- Sensitive store в desktop shell переведен с детерминированного host-derived секрета на app-scoped секрет, защищенный Windows DPAPI для текущего пользователя.
- Legacy plaintext sensitive stores больше не остаются в открытом виде после чтения: при первом успешном чтении они мигрируются в зашифрованный envelope.
- Добавлен отдельный standalone-режим `updater` с пошаговым secure-flow экраном: обнаружение релиза, сравнение версий, проверка доверия, скачивание, сверка сумм и безопасный handoff на установку.
- Для проверки UX и безрискового просмотра потока добавлен `preview`-режим апдейтера, который проходит все этапы без установки файлов и доступен для ручного запуска.
- В релизный и установочный bundle добавлен отдельный `cerbena-updater.exe`: его можно запускать вручную как самостоятельный апдейтер, и при актуальной версии он корректно сообщает, что обновление не требуется.
- Sync E2E больше не использует детерминированный placeholder для wrapped key: payload теперь шифруется случайным data key, а сам data key оборачивается через AEAD с профильным и key-id контекстом.
- Transport guard для sync переведен на структурную проверку TLS-версий и окно recent nonces вместо сравнения строк и одного последнего nonce, что закрывает downgrade/replay слабости модели.
- Remote DNS blocklists теперь по умолчанию требуют `https`, умеют проверять ожидаемый `SHA-256` для curated-источников и fail-closed отклоняют небезопасные схемы или mismatch по целостности.
- Распаковка `tar.gz` для network runtime больше не доверяет library-default unpack: пути нормализуются вручную, traversal (`..`, absolute path) блокируется, а archive links отклоняются до записи на диск.
- Сохранена обратная совместимость sync E2E: новые зашифрованные envelope продолжают читать legacy-полезные нагрузки через версионированный decrypt fallback.
- Standalone updater стал компактнее и чище: уменьшено окно, конвейер переведен в вертикальную колонку, выровнены отступы, убраны редактируемые поля версий, весь пошаговый текст переведен на общий RU/EN i18n слой, а кнопка закрытия переведена на backend-команду без ошибки `window.close not allowed`.
- Повторяющееся предупреждение о поврежденном `traffic_gateway_log.json` устранено: corrupt-файл теперь переносится в quarantine/backup и удаляется из активного пути, поэтому сообщение не повторяется на каждом старте.
- Режим `full` в release pipeline больше не маскирует secret/vulnerability-сканы как пропущенные при наличии локального Docker sandbox: `gitleaks` и `trivy` теперь считаются завершенными только после фактического прогона sandbox-скрипта внутри vulnerability gates.

### Расширения

- Во вкладке `Расширения` добавлен выпадающий список `Список расширений` с фильтрами `Все`, `Chromium`, `Firefox` и `Гибридный`.
- Карточки расширений теперь фильтруются по движку без перезагрузки страницы.
- Для импортируемых пакетов по-прежнему сохраняются логотип, версия, движок, назначения профилям и флаги автообновления и защиты от очистки.
- Добавлены папочные действия `Импорт` и `Экспорт` с режимами `Файл` и `Архив`: файловый режим переносит manifest со ссылками и метаданными, а архивный собирает отдельную папку с manifest и пакетами расширений.
- Файловый экспорт и импорт библиотеки расширений теперь дополнительно переносят ручные теги вместе со ссылками и остальными метаданными.
- Для расширений добавлены ручные теги, показ только первого тега на карточке и фильтр по тегам в правой части списка без чувствительности к регистру.
- Поле тегов переведено на общий dropdown-механизм для расширений и профилей, поэтому ввод больше не теряет фокус на каждом символе и позволяет и выбирать существующие теги, и создавать новые.

### Стабильность

- Добавлены тестовые проверки для одиночных и составных ZIP-архивов расширений.
- Обновлены контрактные версии в тестах и runtime-значениях, чтобы они соответствовали релизной линии `1.0.2`.

### Документация

- Обновлены `README`, `README.en.md`, `ui/desktop/README.md` и двуязычная wiki: документация теперь явно фиксирует, что проект обеспечивает изоляцию профилей, защищенное хранение чувствительного состояния desktop shell, доверенный standalone-апдейтер с `checksums.sig`/`SHA-256` проверками и обратную совместимость sync E2E после hardening.
- Release runbook и docs-навигация дополнены описанием `cerbena-updater.exe`, `checksums.sig`, preview-режима апдейтера и требований к security-проверкам перед выпуском.
- В `README` и `README.en.md` добавлена ссылка на публичную wiki, русская документация дочищена от смешанных английских фрагментов, а docs-quality тест расширен на все файлы `docs/ru/**/*.md`.

### Установщик и хранение данных

- Desktop launcher и связанные локальные хранилища на Windows теперь целенаправленно используют папку `%LOCALAPPDATA%\\Cerbena Browser` вместо dev-ориентированного app data пути.
- Tauri `productName` выровнен с именем `Cerbena Browser`, чтобы путь пользовательских данных и имя установленного продукта совпадали с установщиком.
- Для `npm run dev` и других debug-сборок на Windows возвращен отдельный путь `%LOCALAPPDATA%\\dev.browser.launcher`, чтобы разработческий runtime не смешивался с установленным приложением.
