title: Маршрутизация и маршрутный runtime

Сетевой контур Cerbena Browser строится вокруг профильной политики маршрутизации, локального шлюза трафика и управляемого launcher runtime маршрута.

## Поддерживаемые режимы маршрута

- `direct`
- `proxy`
- `vpn`
- `tor`
- `hybrid`

Дополнительно доступны глобальные настройки `global VPN` и `block without VPN`.

## Шаблоны соединений

Шаблоны описывают один или несколько узлов:

- `proxy` (`http`, `socks4`, `socks5`, `shadowsocks`, `vmess`, `vless`, `trojan`);
- `vpn` (`wireguard`, `openvpn`, `amnezia`);
- `tor` (`none`, `obfs4`, `snowflake`, `meek`).

Шаблон можно привязать к конкретному профилю или выбрать как глобальный маршрут по умолчанию.

## Стратегии изоляции

Cerbena исполняет маршрут через одну из четырех явных стратегий:

- `isolated` — маршрут остается внутри локального шлюза и runtime в пользовательском режиме.
- режим нативной совместимости — разрешается устаревший путь, совместимый с нативным режимом, если шаблон нельзя безопасно провести через чистый пользовательский режим.
- контейнерный режим — маршрут запускается внутри отдельной контейнерной среды профиля.
- `blocked` — запуск запрещен, потому что запрошенный маршрут и выбранная политика изоляции несовместимы.

Неявных резервных переключений между этими стратегиями больше нет. Launcher хранит выбранную политику в отдельном хранилище сетевой изоляции и при обновлении устаревших профилей может автоматически перевести их в явный режим совместимости, а не менять сетевое поведение скрыто.

## Глобальная и профильная изоляция в UI

Launcher теперь разделяет две разные задачи:

- экран `VPN` управляет глобальным маршрутом по умолчанию и фреймом `Сетевая изоляция профилей` для глобального VPN-режима;
- вкладка `VPN` в модалке профиля управляет шаблоном конкретного профиля и фреймом `Сетевая изоляция профиля` именно для этого маршрута.

Если глобальный VPN выключен, глобальный фрейм изоляции скрывается, потому что в этом месте нет активного глобального маршрута, который нужно анализировать.

## Route runtime

Когда маршрут нельзя провести как обычное прямое соединение, Cerbena поднимает управляемый локальный runtime:

- `sing-box` для цепочек, `v2ray`-совместимых транспортов и маршрутов `wireguard`/`amnezia` в пользовательском режиме;
- `openvpn` для `openvpn`;
- `tor` и подключаемые транспортные модули для `TOR`.

Runtime стартует и останавливается вместе с жизненным циклом профиля. Обычный профильный путь не должен поднимать постоянный общесистемный туннель: трафик профиля идет через локальный шлюз и его маршрутный runtime, а launcher при старте и удалении дополнительно вычищает устаревшие хвосты `AmneziaWG` от старых установок.

## Контейнерная изоляция маршрутов

Контейнерный режим использует управляемый launcher вспомогательный образ и отдельную сеть контейнерной среды на профиль.

Что делает launcher в этом режиме:

- проверяет доступность локальной контейнерной среды до запуска;
- ограничивает число активных слотов изоляции;
- резервирует для каждого профиля отдельную управляемую сеть;
- при первом запуске собирает вспомогательный образ;
- поднимает профильный вспомогательный runtime внутри изолированной среды;
- удаляет управляемые launcher контейнеры, сети и вспомогательные образы при очистке и удалении приложения.

Внутри контейнерной изоляции теперь поддерживаются:

- совместимые `proxy`-шаблоны;
- шаблоны `V2Ray/XRay`;
- `WireGuard` и `Amnezia` в пользовательском режиме;
- одноузловой `OpenVPN`;
- `TOR` с `obfs4`, `snowflake` и `meek_lite`-совместимыми сценариями мостов.

Для `AmneziaWG`, который требует только нативный режим, Cerbena тоже держит туннель внутри контейнерной среды конкретного профиля: launcher запускает там `amneziawg-go`, `awg-quick` и локальный `SOCKS`-шлюз вместо того, чтобы переводить весь хост в общесистемный туннель.

Текущая ревизия вспомогательного образа — `2026-05-02-r5`. Встроенный `sing-box` теперь собирается с `with_utls`, поэтому контейнерный `VLESS Reality` больше не должен падать на отсутствии `uTLS`.

## Traffic gateway

Локальный шлюз:

- принимает профильный трафик;
- применяет kill-switch, DNS-, сервисные и доменные правила, а также пользовательские блокировки;
- пишет журнал решений во вкладку `Трафик`;
- маршрутизирует соединение через runtime `SOCKS5`-узел или напрямую через настроенный proxy.

Для надежности этого контура были важны такие исправления:

- буферизированные TLS-байты сохраняются для `CONNECT`-туннелей;
- короткие таймауты рукопожатия снимаются перед долгоживущим мостом;
- Windows-сокеты возвращаются в блокирующий режим перед запуском долгого сетевого моста.

Именно эти исправления убрали ситуации, когда контейнерные маршруты рвались на `ERR_CONNECTION_CLOSED` или Windows `os error 10035` сразу после разрешенного `CONNECT`.

## Kill-switch

Kill-switch блокирует трафик, если:

- обязательный VPN/runtime-маршрут недоступен;
- `TOR`-шаблон невалиден или нужный мост недоступен;
- глобальная политика требует VPN, но активной runtime-сессии нет.

Ни один следующий слой политики не должен ослаблять это правило.

## Что должно чиститься

Если меняется управляемый launcher сетевой или runtime-контур, в том же цикле должна обновляться очистка. Для текущего стека это означает, что деинсталлятор или фоновая очистка обязаны удалять:

- профильные локальные слушатели шлюза;
- route runtime-каталоги и конфигурационные хвосты;
- данные профилей, каталоги подготовки обновлений, пакеты расширений, защищённый системным шифрованием секрет приложения и сохранённые JSON-хранилища внутри локального каталога состояния приложения;
- устаревшие службы `AmneziaWGTunnel$awg-*`, если они принадлежат Cerbena;
- артефакты хранилища сетевой изоляции;
- вспомогательные контейнеры, сети контейнерной среды и вспомогательные образы, созданные для контейнерной изоляции.

## Что проверять при проблемах

1. Выбран ли корректный шаблон.
2. Какая стратегия реально разрешена: изолированный режим, режим нативной совместимости, контейнерный режим или блокировка запуска.
3. Поднят ли route runtime.
4. Не включен ли глобальный `block without VPN`.
5. Есть ли явная причина блокировки в `Трафике` или в operator/runtime diagnostics.
