Cerbena Browser умеет подготавливать и использовать управляемые runtime-бинарники как для сетевого контура, так и для контейнерной изоляции трафика.

## Что сюда входит

- браузерные runtime для `Chromium`, `ungoogled-chromium` и `LibreWolf`;
- маршрутные backend-компоненты `sing-box`, `openvpn`, `amneziawg` и `tor`;
- вспомогательные артефакты контейнерной среды для сетевой изоляции отдельного профиля;
- локальное кеширование, проверка целостности и повторное использование артефактов.


## Операторские задачи

- проверять, что подготовка бинарных файлов и runtime-компонентов завершилась успешно;
- отслеживать ошибки route runtime через operator/runtime-диагностику и профильные runtime-журналы;
- понимать, какой backend и какая стратегия изоляции были выбраны для конкретного шаблона профиля;
- контролировать, что вспомогательные контейнеры, выделенные сети контейнерной среды и очистка вспомогательного образа синхронизированы с поведением деинсталлятора launcher.
- проверять, что деинсталляция и служебная очистка не оставляют launcher-управляемых следов от старых browser/runtime-артефактов и работают только с актуальными путями `chromium`/`ungoogled-chromium`/`librewolf`.

## Практические замечания по runtime

- Вспомогательный образ контейнерной среды управляется launcher и собирается при первом использовании.
- Текущая ревизия вспомогательного образа — `2026-05-02-r5`.
- Для `VLESS Reality` и других транспортных сценариев, которым нужен `uTLS`, вспомогательный образ теперь содержит `sing-box`, собранный с `with_utls`.
- Для `AmneziaWG`, который требует только нативный режим, Cerbena держит режим совместимости внутри отдельного контейнера профиля через `amneziawg-go`, `awg-quick` и локальный `SOCKS`-узел вместо требования к общесистемной VPN-сессии на хосте.
