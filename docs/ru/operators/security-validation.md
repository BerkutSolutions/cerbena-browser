Эта страница служит операторской финальной проверкой для блока задач `TASKS4`, который закрывает `U14-2`.

Она нужна затем, чтобы Cerbena Browser не делала завышенных заявлений о защите только по факту наличия кода или интерфейса.

## Матрица проверок

| Область | Реализованные меры | Автоматические подтверждения | Примечания |
|---|---|---|---|
| Изоляция рабочей области и сессий | брокер сессий запуска, отпечаток рабочей области, маркерный файл сессии, доверенное повторное подключение только через backend | тесты в `ui/desktop/src-tauri/src/launch_sessions.rs`, `cargo test --workspace` | Негативные сценарии включают подмену маркерного файла и отклонение несоответствия движка. |
| Чувствительное состояние на диске | зашифрованные хранилища launcher для sync, маршрутизации ссылок, сессий запуска и глобального реестра безопасности | тесты в `ui/desktop/src-tauri/src/sensitive_store.rs`, тесты в `ui/desktop/src-tauri/src/launcher_commands.rs` | Старый открытый формат поддерживается только как миграция для чтения. |
| Защищенный запуск профилей | обязательная разблокировка, запрет опасных сочетаний с доступом к системе и `KeePassXC`, ограничения расширений для политики `maximum` | тесты в `ui/desktop/src-tauri/src/profile_security.rs`, `cargo test --workspace` | Интерфейс не является границей безопасности; решение принимает backend. |
| Защита cookie и чувствительных поверхностей | копирование cookie требует корректной разблокировки и не допускает участие защищенных профилей | тесты в `ui/desktop/src-tauri/src/profile_commands.rs` | Для незаблокированных и незащищенных профилей копирование cookie по-прежнему возможно по дизайну. |
| Реакции на состояние устройства | backend-проверка сигналов хоста с реакциями `allow / warn / confirm / refuse` | тесты в `ui/desktop/src-tauri/src/device_posture.rs` | Это не антивирус, а контур оценки состояния хоста для политики запуска. |
| Защищенные снимки | зашифрованное создание и восстановление профильного снимка с проверкой целостности | тесты в `ui/desktop/src-tauri/src/sync_snapshots.rs`, тесты sync-команд | Заглушка `btoa("snapshot")` больше не используется. |
| Аварийная очистка и защищенные сайты | остановка отслеживаемого процесса, выборочные цели очистки, объединение путей защищенных сайтов, точная очистка Chromium/Firefox `SQLite`, внешний сигнальный оверлей | тесты в `internal/profile/src/storage.rs`, `internal/profile/tests/profile_manager_tests.rs`, тесты в `ui/desktop/src-tauri/src/launcher_commands.rs` | Точное сохранение теперь покрывает `cookie`/историю в `SQLite`-хранилищах Chromium и Firefox-подобных движков. |

## Обязательные проверки перед выпуском

Минимальный набор проверок для этого блока:

- `cargo test --workspace`
- `cd ui/desktop && npm test`
- `npm run docs:build`
- тесты качества документации и тесты стабильности launcher
- ручная проверка остаточных рисков ниже перед любыми заметками о выпуске с акцентом на защиту

## Остаточные риски и честные ограничения

Cerbena Browser теперь сильнее контролирует границу launcher, но не заявляет:

- иммунитет к скомпрометированному хосту;
- изоляцию уровня гипервизора или `Qubes OS`;
- абсолютную защиту от инъекций, если враждебный процесс уже контролирует пользовательскую сессию;
- криминалистически идеальное стирание всех артефактов браузера за пределами явно покрытых хранилищ `cookie`/истории в `SQLite`.

## Что должен понимать оператор

- Аварийную очистку нужно воспринимать как сильную очистку под управлением launcher, а не как гарантированное криминалистически надежное стирание.
- Проверку состояния устройства нужно воспринимать как вход в политику запуска, а не как полноценный продукт класса обнаружения угроз на конечной точке.
- Шифрование состояния launcher защищает от простого локального чтения на диске, но не отменяет необходимость защищать сам хост и учетную запись пользователя.
- Точное сохранение защищенных сайтов теперь действует для хранилищ `cookie`/истории в `SQLite` у Chromium и Firefox-подобных движков, но не должно распространяться на пароли, базы закладок или произвольные хранилища расширений.
