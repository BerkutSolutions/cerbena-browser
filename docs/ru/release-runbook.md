## Базовый цикл

1. Выполнить `cargo test --workspace`.
2. Выполнить проверки документации, `i18n`, UI smoke и локальную предварительную проверку.
3. Подготовить материалы подписи релиза и переменные окружения оператора.
4. Собрать релизные артефакты и установщик.
5. Проверить `README`, двуязычную wiki, `CHANGELOG.md` и матрицу трассировки требований.

Перед выполнением упаковки релиза обязательно прочитать:

- [Модель доверия релизов](./release-trust.md)

## Команды

### Подготовка самоподписанных материалов подписи

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\new-release-signing-material.ps1
```

Скрипт создает локальный набор для оператора:

- приватный ключ в формате `XML` для отдельной подписи контрольных сумм;
- публичный ключ в формате `XML` для коммита в `config/release/release-signing-public-key.xml` при ротации;
- самоподписанный `Authenticode PFX`;
- `PEM` публичного сертификата;
- пароль `PFX`.

### Быстрая предварительная проверка

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\local-ci-preflight.ps1 -CompactOutput
```

Локальная предварительная проверка теперь за один проход прогоняет проверку контейнерной среды, проверки безопасности и проверки уязвимостей.

### Полная релизная упаковка

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\release.ps1 -Mode package -CompactOutput
```

Перед упаковкой должны быть заданы:

- `CERBENA_RELEASE_SIGNING_PRIVATE_KEY_XML` или `CERBENA_RELEASE_SIGNING_PRIVATE_KEY_PATH`
- `CERBENA_AUTHENTICODE_PFX_PATH`
- `CERBENA_AUTHENTICODE_PFX_PASSWORD`
- опционально `CERBENA_AUTHENTICODE_TIMESTAMP_URL`

### Сборка Windows-установщика

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\build-installer.ps1
```

### Локальная проверка отдельной подписи и Windows-подписей

```powershell
. .\scripts\release-signing.ps1
Verify-ReleaseChecksumSignature `
  -ChecksumsPath .\build\release\<version>\checksums.txt `
  -SignaturePath .\build\release\<version>\checksums.sig

Verify-WindowsArtifacts @(
  '.\build\release\<version>\staging\cerbena-windows-x64',
  '.\build\installer\<version>\output\cerbena-browser-<version>.msi',
  '.\build\installer\<version>\output\cerbena-browser-setup-<version>.exe'
)
```

## Что должно лежать в релизе

В релизах GitHub рекомендуется публиковать:

- `cerbena-browser-<version>.msi` как основной установщик Windows и артефакт для передачи в средство обновления;
- `cerbena-browser-setup-<version>.exe` как ручной совместимый резервный установщик;
- `cerbena-windows-x64.zip` как переносимый архив релизной сборки;
- `cerbena-updater.exe` как отдельный standalone-апдейтер;
- `checksums.txt`;
- `checksums.sig`;
- `release-manifest.json`.

`MSI` теперь собирается локально, подписывается тем же материалом `Authenticode`, что и остальные Windows-артефакты, и должен публиковаться в GitHub release как основной пользовательский артефакт установки. Установщик `.exe` сохраняется как совместимый резервный путь для ручного восстановления и сценариев, где запуск через `msiexec` не подходит.

Граница ответственности средства обновления для Windows теперь зафиксирована явно:

- установленная Windows-сборка предпочитает подписанный `MSI`;
- переносимая Windows-сборка сохраняет подписанный `.zip` как основной путь обновления;
- если средство обновления выбирает `MSI`, оно не извлекает внутреннее содержимое и не переносит доверие на непрозрачную перепаковку после загрузки;
- вместо этого после проверки контрольной суммы и подписи средство обновления передает в `msiexec` сам проверенный `MSI`, а состояние перезапуска и очистки записывает в локальное хранилище для аудита.

## Перед выпуском

- проверить ручной сценарий обновления, отдельную подпись контрольных сумм и подписи `Authenticode` у артефактов;
- проверить сценарий обновления через `MSI` для установленной Windows-сборки: проверенную загрузку, передачу в `msiexec`, перезапуск и восстановление после отмены или сбоя;
- убедиться, что standalone-апдейтер корректно показывает состояние `version is current` для актуальной сборки и проходит `preview`-сценарий без установки;
- убедиться, что `sync`, `route runtime`, traffic gateway и поток установщика не содержат известных регрессий;
- убедиться, что документация отражает актуальный UI, релизные скрипты, способы установки и ограничения самоподписанного доверия.

## Откат

1. Остановить активный экземпляр.
2. Вернуть предыдущий подписанный артефакт.
3. Проверить консистентность метаданных профилей.
4. Повторно прогнать ключевые smoke-проверки.

## Политика восстановления для MSI-обновлений

Если проверенный `MSI` был скачан, но установка завершилась неуспешно, действуют следующие правила:

1. Скачанный пакет не считается успешно примененным, пока `msiexec` не завершился успешным кодом.
2. Предыдущая установленная версия остается авторитетным рабочим состоянием, пока транзакция `MSI` не завершена.
3. Если `msiexec` вернул отмену (`1602`), обновление помечается как отмененное, ожидающее состояние сбрасывается, а повтор допускается только после нового решения оператора.
4. Если `msiexec` вернул код `1618`, нужно дождаться завершения другого потока установщика Windows и затем повторить обновление.
5. Если локальное хранилище обновления зависло в `applied_pending_relaunch`, но текущая версия уже совпадает с подготовленной, состояние нужно примирить обратно к `up_to_date`.

Операторское восстановление при ошибках публикации или материала доверия:

- сломанная публикация `MSI`: удалить ошибочный артефакт из релиза GitHub, заново собрать и подписать `MSI`, затем вместе перевыпустить `checksums.txt`, `checksums.sig` и `release-manifest.json`;
- ротированный или отозванный самоподписанный материал: выпустить новую пару ключей, закоммитить новый публичный `XML`, пересобрать релизные артефакты с новым приватным материалом и не переиспользовать старый секрет;
- несовпадение контрольной суммы или подписи: немедленно остановить распространение, пересобрать выпуск из чистой рабочей копии и не помечать поврежденные артефакты как доверенные.

## После релиза

- обновить `CHANGELOG.md`;
- зафиксировать проверенный набор контрольных проверок качества и безопасности;
- при необходимости обновить сведения в матрице трассировки требований;
- приложить в карточку релиза на GitHub и `MSI`, и резервный установщик `.exe`.
