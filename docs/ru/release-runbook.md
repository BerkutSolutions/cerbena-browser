## Базовый цикл

1. Выполнить `cargo test --workspace`.
2. Выполнить проверки документации, `i18n`, UI smoke и локальную предварительную проверку.
3. Собрать релизные артефакты и установщик.
4. Проверить `README`, двуязычную wiki, `CHANGELOG.md` и матрицу трассировки требований.

## Команды

### Быстрая предварительная проверка

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\local-ci-preflight.ps1 -CompactOutput
```

Локальная предварительная проверка теперь за один проход прогоняет проверку контейнерной среды, проверки безопасности и проверки уязвимостей.

### Полная релизная упаковка

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\release.ps1 -Mode package -CompactOutput
```

### Сборка Windows-установщика

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\build-installer.ps1
```

## Что должно лежать в релизе

В релизах GitHub рекомендуется публиковать:

- `cerbena-browser-setup-<version>.exe` как основной установщик Windows;
- `cerbena-windows-x64.zip` как переносимый архив релизной сборки;
- `cerbena-updater.exe` как отдельный standalone-апдейтер;
- `checksums.txt`;
- `checksums.sig`;
- `release-manifest.json`.

Установщик `.exe` собирается локально и должен попадать в релиз как основной пользовательский артефакт установки.

## Перед выпуском

- подтвердить, что `Wayfern`-поток не требует пропущенного подтверждения `ToS`;
- проверить ручной сценарий обновления и подписи артефактов;
- убедиться, что standalone-апдейтер корректно показывает состояние `version is current` для актуальной сборки и проходит `preview`-сценарий без установки;
- убедиться, что `sync`, `route runtime`, traffic gateway и поток установщика не содержат известных регрессий;
- убедиться, что документация отражает актуальный UI, release-скрипты и способы установки.

## Откат

1. Остановить активный экземпляр.
2. Вернуть предыдущий подписанный артефакт.
3. Проверить консистентность метаданных профилей.
4. Повторно прогнать ключевые smoke-проверки.

## После релиза

- обновить `CHANGELOG.md`;
- зафиксировать проверенный набор контрольных проверок качества и безопасности;
- при необходимости обновить сведения в матрице трассировки требований;
- приложить установщик `.exe` в карточку релиза на GitHub.
