## Назначение

Cerbena сейчас использует **внутреннюю модель доверия релизов на самоподписанной подписи** для одного сопровождающего.
Она нужна для трех задач:

- подтверждать целостность и происхождение опубликованных артефактов;
- принудительно проверять метаданные релиза в модуле обновления и релизных скриптах;
- разделять публичные данные проверки и приватные материалы подписи.

Эта модель **не равна публичному доверию Windows**. Корректная самоподписанная подпись не означает:

- отсутствие предупреждений `SmartScreen`;
- отсутствие предупреждений браузеров и системы загрузки;
- публичную репутацию, эквивалентную `OV/EV` сертификату подписи кода.

## Авторитетная граница доверия

Для текущего этапа авторитетная граница доверия в Cerbena строится так:

1. `checksums.txt` подписывается отдельной release-подписью.
2. Публичный ключ проверки этой подписи коммитится в репозиторий:
   - `config/release/release-signing-public-key.xml`
3. Модуль обновления принимает релизный артефакт только после:
   - успешной проверки `checksums.sig`;
   - успешного совпадения `SHA-256` для выбранного файла релиза.
4. Windows-исполняемые артефакты дополнительно подписываются подписью `Authenticode` через операторский самоподписанный `PFX`.
5. Генерация релиза завершается ошибкой, если требуемая подпись отсутствует или не проходит проверку.

## Что считается обязательным для подписи

Для Windows действует политика обязательного сбоя на всех публикуемых артефактах, которые участвуют в установке, обновлении или запуске.

Обязательная подпись `Authenticode` требуется для:

- основного приложения `cerbena.exe`;
- отдельного обновлятора `cerbena-updater.exe`;
- launcher `cerbena-launcher.exe`;
- любого `*.exe`, `*.dll`, `*.sys`, `*.msi`, попавшего в публичный комплект релиза;
- любого `*.exe` или `*.msi`, опубликованного как Windows-установщик.

Обязательная отдельная подпись релиза требуется для:

- `checksums.txt`;
- `checksums.sig`.

`release-manifest.json` и архивы вроде `cerbena-windows-x64.zip` проверяются через подписанный `checksums.txt`.

## Политика по вспомогательным бинарным файлам

Неподписанные вспомогательные двоичные файлы не допускаются.

Если в комплект релиза или каталог выходных файлов установщика попадает новый Windows-вспомогательный бинарный файл из списка:

- `*.exe`
- `*.dll`
- `*.sys`
- `*.msi`

и он не подписан ожидаемым релизным сертификатом, релизный поток должен завершиться ошибкой.
Переходного предупредительного режима для таких файлов нет.

## Публичное и приватное

В репозитории допускается хранить только публичные данные проверки:

- `config/release/release-signing-public-key.xml`

Приватный материал подписи должен подаваться только на машине оператора:

- `CERBENA_RELEASE_SIGNING_PRIVATE_KEY_XML` или `CERBENA_RELEASE_SIGNING_PRIVATE_KEY_PATH`
- `CERBENA_AUTHENTICODE_PFX_PATH`
- `CERBENA_AUTHENTICODE_PFX_PASSWORD`
- опционально `CERBENA_AUTHENTICODE_TIMESTAMP_URL`

## Операторский запуск и локальная проверка

Новый набор самоподписанных материалов на машине оператора создается командой:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\new-release-signing-material.ps1
```

Минимальный порядок действий оператора:

1. Сгенерировать ключевую пару и самоподписанный `PFX`.
2. Вынести приватные файлы из рабочей копии репозитория в безопасное локальное хранилище.
3. При ротации коммитить только новый публичный `XML`.
4. Перед упаковкой задать `CERBENA_RELEASE_SIGNING_PRIVATE_KEY_PATH`, `CERBENA_AUTHENTICODE_PFX_PATH` и `CERBENA_AUTHENTICODE_PFX_PASSWORD`.
5. Выполнять релизную упаковку только на этой операторской машине.

Локальная проверка после сборки:

```powershell
. .\scripts\release-signing.ps1
Verify-ReleaseChecksumSignature `
  -ChecksumsPath .\build\release\<version>\checksums.txt `
  -SignaturePath .\build\release\<version>\checksums.sig

Verify-WindowsArtifacts @(
  '.\build\release\<version>\staging\cerbena-windows-x64',
  '.\build\installer\<version>\output\cerbena-browser-<version>.msi',
  '.\build\installer\<version>\output\cerbena-browser-setup-<version>.exe'
)
```

Закоммиченного публичного `XML` достаточно для проверки отдельной подписи релиза в чистой копии репозитория, но он не дает возможности подписывать новые артефакты.

## Ротация и будущая миграция

Ротация выполняется через:

- `scripts/new-release-signing-material.ps1`

Скрипт генерирует:

- новый приватный ключ в формате `XML` для подписи `checksums.sig`;
- новый публичный ключ в формате `XML`;
- новый самоподписанный `Authenticode PFX`;
- публичный сертификат `PEM`.

После ротации нужно:

1. безопасно вынести приватные файлы из рабочей директории;
2. заменить `config/release/release-signing-public-key.xml`;
3. выпускать новые релизы только с новой парой ключей;
4. не заявлять в документации, что самоподписанная подпись решает репутацию Windows.

Переход на будущий `OV/EV` сертификат должен сохранить ту же модель:

- проверка контрольных сумм в модуле обновления остается обязательной;
- `Authenticode` усиливает встроенный в Windows слой доверия, но не заменяет отдельную проверку релизных подписей.

## Ограничения доверия и публичная миграция

Текущая самоподписанная модель сознательно ограничена:

- она подтверждает целостность и операторское происхождение артефактов внутри потока Cerbena;
- она не дает репутацию `SmartScreen`;
- она не убирает браузерные предупреждения сама по себе;
- ее нельзя описывать как эквивалент публичному `OV/EV` сертификату.

Будущая миграция на публичный сертификат подписи кода должна сохранить:

- обязательную отдельную проверку `checksums` для модуля обновления;
- ту же модель публикации `release-manifest.json` и контрольных сумм;
- операторские цепочки проверки и восстановления, которые остаются валидными даже при смене типа `Authenticode`.
